## 1. コスト削減のベストプラクティス一覧

### 1-1. クエリレベルのコスト削減

#### `SELECT *` を避ける

BigQuery は列指向データベースであり、**スキャンした列のデータ量に基づいて課金**される。`SELECT *` はすべての列を読み込むため、不要なコストが発生する。

```sql
-- NG: 全列スキャン
SELECT * FROM `project.dataset.sales`;

-- OK: 必要な列のみ指定
SELECT order_id, customer_id, amount
FROM `project.dataset.sales`;
```

#### `LIMIT` はコスト削減にならない

`LIMIT` 句による行制限はデータスキャン後に適用されるため、**スキャンコストは変わらない**。

#### `WHERE` 句でデータを絞り込む

スキャン対象データを減らす最も基本的な手法。パーティション列に対するフィルタが最も効果的。

---

### 1-2. テーブル設計によるコスト削減

#### パーティショニング (Partitioning)

テーブルを日付などの列で分割し、クエリが**必要なパーティションのみをスキャン**するように誘導する。

```sql
-- パーティション列でフィルタすることでコストを劇的に削減
SELECT *
FROM `project.dataset.events`
WHERE DATE(event_timestamp) = '2025-03-01';
```

#### クラスタリング (Clustering)

パーティション内のデータを指定列でソートして格納する。フィルタ・集計時に不要なデータブロックをスキップできる。

```sql
CREATE TABLE `project.dataset.sales`
PARTITION BY DATE(created_at)
CLUSTER BY region, product_id
AS SELECT * FROM source_table;
```

Google Cloud は 2025年より**パーティショニング・クラスタリング推奨機能** (GA) を提供しており、最適化の機会を自動的に提示する。

### 1-4. 料金モデルの選択: BigQuery Editions

2025年において最も影響の大きいコスト変数の一つが **BigQuery Editions** の活用である。

| エディション | 特徴 | 向いているワークロード |
|---|---|---|
| Standard | ベースライン、最低コスト | 低負荷・開発環境 |
| Enterprise | クロスプロジェクトスロット共有、最大40%コミット割引 | 安定したプロダクション |
| Enterprise Plus | 最大SLA、マルチリージョン | ミッションクリティカル |

ある事例では、特定ワークロードをオンデマンドから Editions へ切り替えることで **月額 $60,000 → $19,000 への削減** を達成した。

#### ゼロベースライン戦略 (2025年新手法)

Editions ではベースラインスロット数を `0` に設定し、オートスケールのみで運用することで、実質オンデマンドに近い柔軟な課金が可能。

### 1-5. dbt インクリメンタルモデルによるコスト削減

dbt の `incremental` モデルを使うことで、テーブル全体の再構築を避け、**新規・更新されたデータのみを処理**できる。

```sql
{{ config(
    materialized='incremental',
    incremental_strategy='insert_overwrite',
    partition_by={'field': 'event_date', 'data_type': 'date'},
    cluster_by=['user_id']
) }}

SELECT *
FROM {{ source('raw', 'events') }}
{% if is_incremental() %}
WHERE event_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 3 DAY)
{% endif %}
```

大規模テーブルで**100〜200倍のコスト削減**が報告されている。

## 2. パフォーマンス最適化のテクニック

### 2-1. マテリアライズドビュー (Materialized Views)

頻繁に実行される重いクエリの結果を事前計算して保存する。数十億行スキャンが必要なクエリを **数MB のスキャンに圧縮**できる。

```sql
CREATE MATERIALIZED VIEW `project.dataset.mv_daily_revenue`
AS
SELECT
  DATE(created_at) AS date,
  region,
  SUM(amount) AS total_revenue
FROM `project.dataset.orders`
GROUP BY 1, 2;
```

### 2-2. BI Engine によるキャッシング

頻繁に利用するデータをメモリ内にキャッシュし、インメモリ速度でのクエリ実行を実現。Looker Studio や Looker などのダッシュボードワークロードに特に効果的。

### 2-3. INFORMATION_SCHEMA を使ったモニタリング

```sql
-- 過去7日間の高コストクエリTOP10
SELECT
  job_id,
  user_email,
  total_bytes_processed,
  ROUND(total_bytes_processed / POW(10, 12) * 5, 2) AS estimated_cost_usd,
  query
FROM `region-us`.INFORMATION_SCHEMA.JOBS
WHERE creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
ORDER BY total_bytes_processed DESC
LIMIT 10;
```

## 3. 2025年以降の新しい最適化手法の要約

### 3-1. Gemini AI による自動最適化支援

- **SQL・Python コード補完・生成**: 自然言語からクエリを自動生成
- **クエリ解説機能**: 既存の複雑な SQL を平易な言語で説明
- **推奨機能**: パーティショニング、クラスタリング、ビュー作成を自動提案
- **AI クエリプランナー最適化**: `AI.IF`, `AI.CLASSIFY`, `AI.SCORE` 関数のフィルタ内での実行順序をプランナーが自動最適化し、Gemini API コール数を削減

### 3-2. パーティショニング・クラスタリング推奨機能 (GA)

テーブルのクエリパターンを自動分析し、最適なパーティションキーとクラスタリング列を推奨するシステムが GA に昇格。

### 3-3. 物理ストレージ課金モデルへの移行

ストレージの課金モデルを**論理バイト課金から物理（圧縮）バイト課金**に切り替えることで、BigQuery の高圧縮格納のメリットを直接コスト削減に活かせる。
